# Phase 3FA-C ŌĆö Owner-local KIS Similarity Smoke Harness, Disabled by Default v0.1

## 1. Purpose

This phase adds a disabled-by-default owner-local KIS similarity smoke harness scaffold. The
harness provides a safe, deterministic, non-executing boundary that can later be extended for
owner-local manual smoke execution only after separate owner approval. It does not run KIS, does
not run the deterministic similarity engine against live data, does not change the existing
`/api/chart-ai/similarity` route, and does not enable route success.

## 2. Current State

- Phase 3FA-B defined the owner-local KIS similarity smoke plan (design-only, no smoke executed,
  `enabled: false` by default).
- Phase 3FA-A defined the owner-local KIS-normalized similarity execution plan; no execution is
  enabled.
- Phase 3EZ-C added the disabled `/api/chart-ai/similarity` route shell, which always returns
  `feature_disabled` / `feature-flag-off`.
- Phase 3EZ-B added the storage-agnostic usage storage design.
- Phase 3EZ-A added the provider-agnostic real-auth integration design.
- Phase 3EY-A/3EY-B added the server-only KIS OHLC provider contract foundation and its mocked
  adapter.
- Phase 3EX-E-HF1 fixed `/chart-ai` into a real tagged post-it/page-swap UI; no backend/runtime
  change was made.
- No real auth runtime exists.
- No usage storage runtime exists.
- No DB/cache runtime exists.
- No live KIS Chart Similarity execution exists.
- No owner-local KIS similarity smoke has been executed.
- No disabled harness scaffolding exists yet prior to this phase.
- The route remains feature disabled.

## 3. Harness Boundary

- This phase is scaffold/foundation only.
- No KIS call is made by this harness.
- No live similarity engine execution occurs.
- No `/api/chart-ai/similarity` route change is made.
- No `/chart-ai` UI change is made.
- No `process.env` or `.env` value is read.
- No credential is echoed.
- No raw KIS payload is referenced or persisted.
- No actual market value appears in any harness report.
- No public execution is allowed.
- No beta execution is allowed.
- Route success remains disabled by static policy.
- The harness is enabled only if a future, separately approved phase sets every required
  permission flag explicitly to `true`.

## 4. Disabled Harness Flow

The disabled harness proceeds through the following ordered steps, none of which execute a live
provider, live engine, or route call:

1. `load_smoke_plan` ŌĆö load the Phase 3FA-B owner-local smoke plan as a safe design reference.
2. `check_harness_policy` ŌĆö evaluate the harness policy and confirm it remains disabled by default.
3. `verify_route_remains_disabled` ŌĆö confirm `/api/chart-ai/similarity` route success stays
   disabled by static policy.
4. `verify_redaction_policy` ŌĆö confirm the redaction policy prohibits raw payloads, market data,
   and credentials in any report.
5. `verify_no_live_provider` ŌĆö confirm no live KIS provider or client module is imported or called.
6. `verify_no_live_engine` ŌĆö confirm no live similarity engine execution is imported or called.
7. `build_safe_blocked_report` ŌĆö build the deterministic, safe blocked report described in
   Section 5.

## 5. Disabled Report Shape

The disabled harness report includes only these fields:

- `status`
- `smokeId`
- `mode`
- `executedBy`
- `source`
- `providerProbe`
- `normalizationCheck`
- `engineContractCheck`
- `responseRedactionCheck`
- `safeSummary`
- `warnings`

Only `blocked` or `not_run` reports are ever returned by this harness in this phase. No report
generated by this harness may contain a `pass`/success outcome for `providerProbe`,
`normalizationCheck`, or `engineContractCheck`.

## 6. Future Enablement Requirements

A future, separately authorized phase enabling live owner-local smoke execution would require:

- explicit owner approval before any live owner-local smoke;
- a separate harness-enable approval distinct from the owner approval above;
- confirmation of the provider environment (owner-local only, correct KIS credentials configured
  outside of source control);
- confirmation that the redaction policy defined in Phase 3FA-B and reused by this harness remains
  in force;
- confirmation that the `/api/chart-ai/similarity` route remains disabled unless a further,
  separately authorized phase approves enabling it;
- confirmation that no public or beta execution path is enabled as part of live smoke enablement;
- confirmation that no credential or environment value will be echoed in any report produced by
  the enabled harness.

## 7. Validation Strategy

- A committed runtime smoke script exercises the actual harness module (not a re-implementation)
  and asserts: the default policy is disabled; `isOwnerLocalSmokeHarnessEnabled` is `false` under
  the default policy and stays `false` unless every execution permission is explicitly `true`; the
  disabled harness runner never returns a top-level success/`pass` status; the blocked report's
  `providerProbe`, `normalizationCheck`, and `engineContractCheck` fields are `not_run`; no
  serialized harness output contains a user-state field, a token, an email or IP address, a raw
  KIS or auth field, an account/trading field, a market-data field, or a `source=live`/`source=auto`
  literal; and no network call or `process.env` read occurs during harness execution.
- A static checker validates the harness module's and smoke script's source content directly (no
  live KIS/engine/auth/DB/cache import, no `process.env`/`.env`/cookie/header/storage read, no
  `Date.now`/current-date use, no network call), validates the default policy's exact field values,
  validates the exact step/check/report shapes, confirms `/api/chart-ai/similarity`,
  `/chart-ai`, and the deterministic engine files are unchanged from `startingCommit`, and confirms
  only the phase's allowed paths changed.
